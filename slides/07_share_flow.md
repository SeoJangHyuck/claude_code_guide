# 챕터 7: 사내 공유 flow (4슬라이드 / 5분)

> 톤 가이드: 시연(챕터 6)에서 본 skill·rules·agent·hook을 옆자리·팀·전사로 어떻게 넘기는지 4단계로. 비개발자 친화(0~1단계, git 불필요)와 개발자 친화(2단계, git 사용)를 명확히 분리. 마지막 3단계는 전사 표준화.

> 모든 명령어는 2026-05 기준 Claude Code 공식 형식.

---

## 슬라이드 7-1 — 0단계: 파일 한 개로 공유 (45초, git 불필요)

**핵심 메시지**: "skill 1개는 SKILL.md 파일 하나만 메신저로 보내면 끝. git·명령어 다 불필요."

**본문 콘텐츠**:

```
0단계 — 파일 한 개로 공유 (가장 쉬운 방법, git 불필요)

  ───────────────────────────────────────────

  당신:
    .claude/skills/내-skill/SKILL.md 파일을
    메신저·이메일·USB·공유드라이브로 동료에게 전달

  동료:
    자기 프로젝트의 .claude/skills/ 폴더에 그대로 붙여넣기
    → 끝. Claude Code가 자동 인식

  ───────────────────────────────────────────

  📊 적합도
     소요 시간    : 2분
     적합 규모    : 1~2명, git이 어려운 분
     장점        : 가장 직관적, 명령어·git 다 불필요
     단점        : skill 1개씩만 (rules·hook 묶음은 다음 단계)
```

**발표 노트 (45초)**:
> "0단계, 가장 쉬운 방법입니다. git 잘 모르셔도 됩니다. skill 파일 하나 SKILL.md를 메신저·이메일로 동료한테 보내고, 동료는 자기 .claude/skills 폴더에 붙여넣기. 끝. 명령어 한 줄 필요 없습니다. 한계는 한 번에 한 개씩만 가능. 묶음으로 가려면 다음 단계."

**시각 표현 메모**:
- 좌우 흐름 (당신: 파일 첨부 / 동료: 폴더에 복사).
- 하단 적합도 박스.
- "git 불필요" 강조.

## 슬라이드 7-2 — 1단계: 플러그인 zip 공유 (1분 30초, git 불필요)

**핵심 메시지**: "플러그인 = `.claude-plugin/plugin.json` 가진 디렉토리. 만들고 → 로컬 테스트 → zip → 메신저."

**본문 콘텐츠**:

```
1단계 — 플러그인 zip 공유 (묶음, git 불필요)

  ───────────────────────────────────────────

  ① 플러그인 디렉토리 구조 (이름·위치 자유)

  my-plugin/                          ← 플러그인 루트
    ├── .claude-plugin/
    │   └── plugin.json               ← 필수 매니페스트
    ├── skills/<이름>/SKILL.md
    ├── agents/<이름>.md
    └── hooks/hooks.json              ← 선택

  ⚠️ 흔한 실수: skills·agents·hooks를 .claude-plugin/
     안에 넣지 말 것. plugin.json '옆'에 같은 레벨로.

  ───────────────────────────────────────────

  ② plugin.json (4필드면 충분)

  {
    "name": "research-team-plugin",       ← 호출 prefix
    "description": "정책 연구원 업무 자동화 묶음",
    "version": "1.0.0",
    "author": { "name": "BDI 데이터팀" }
  }

  → 동료가 호출할 때:  /research-team-plugin:skill-name
                       (네임스페이스 = name 필드)

  ───────────────────────────────────────────

  ③ 로컬 테스트 → ④ zip 압축 → ⑤ 메신저 전달

  $ claude --plugin-dir ./my-plugin    ← 보내기 전 검증

  ───────────────────────────────────────────

  동료의 설치 (두 줄):

  /plugin marketplace add /풀어둔/경로
  /plugin install research-team-plugin

  ───────────────────────────────────────────

  📊 적합도
     소요 시간    : 30분 (1회성)
     적합 규모    : 3~10명, git이 어려운 팀
     장점        : 묶음 설치, git 서버·계정 불필요, 네임스페이스로 충돌 방지
     단점        : 자동 업데이트 X (수정 시 zip 다시 전달)
```

**발표 노트 (1분 30초)**:
> "1단계, 플러그인 zip 공유입니다. 0단계가 한 파일이었다면 여기는 묶음으로 가요. 핵심 규칙 하나만 — 플러그인은 `.claude-plugin/plugin.json` 매니페스트를 가진 디렉토리입니다. 이름이랑 위치는 자유롭게 정하셔도 되고요.
>
> 만들기는 세 단계입니다. 먼저 폴더 하나(예: `my-plugin`) 만들고, 안에 `.claude-plugin/plugin.json` 하나 두고, `skills`·`agents`·`hooks` 폴더는 `plugin.json` '옆'에 같은 레벨로 둡니다. 흔한 실수가 이 폴더들을 `.claude-plugin/` 안에 넣는 건데, 그건 안 됩니다. 둘째, `plugin.json` 4필드 — `name`이 호출 prefix가 됩니다. `/my-plugin:skill` 이런 식이에요. 셋째, zip 만들기 전에 `claude --plugin-dir`로 로컬에서 한 번 테스트하시고, 정상 동작하면 압축해서 메신저로 보냅니다.
>
> 동료는 zip 풀고 두 줄 — `marketplace add`와 `install`. 끝. 단점은 자동 업데이트가 없다는 것 — 수정하면 zip 재전달. 그게 부담되면 다음 2단계로 갑니다."

**시각 표현 메모**:
- 좌측: 만드는 사람 단계(①~⑤) — 디렉토리 구조 + JSON 코드블록.
- 우측: 받는 사람 단계 — 두 줄 명령 박스 + 네임스페이스 호출 예시.
- "흔한 실수" 경고를 시각적으로 강조 (오렌지 박스).
- 좌측 단계 번호(1) 강조.

**근거 (공식 문서)**:
- 플러그인 정의: `.claude-plugin/plugin.json` 가진 디렉토리 (이름·위치 자유). https://code.claude.com/docs/en/plugins (Plugin structure overview)
- 흔한 실수 경고 원문: "Don't put `commands/`, `agents/`, `skills/`, or `hooks/` inside the `.claude-plugin/` directory."
- 로컬 테스트: `--plugin-dir` 플래그 (zip 압축 파일도 지원, v2.1.128+)
- 권장은 별도 디렉토리(네임스페이싱·마켓플레이스 등록 유리)지만 `.claude/` 자체를 플러그인 루트로 쓰는 것도 기술적으로 가능.

## 슬라이드 7-3 — 2단계: 플러그인 git 공유 (1분 30초, 자동 업데이트)

**핵심 메시지**: "git을 쓸 수 있다면 한 줄 설치 + 자동 업데이트. 개발자 친화."

**본문 콘텐츠**:

```
2단계 — 플러그인 git 공유 (자동 업데이트)

  ───────────────────────────────────────────

  1단계와 같은 플러그인 디렉토리 + git 저장소에 push

  당신:
    $ cd my-plugin/
    $ git init && git remote add origin <git-url>
    $ git add . && git commit -m "share research plugin"
    $ git push -u origin main

  ───────────────────────────────────────────

  동료의 설치 (한 줄):

  /plugin install research-team-plugin@<git-url>

  ───────────────────────────────────────────

  ⭐ 1단계(zip) 대비 차별점
     자동 업데이트  : 당신이 git push 하면 동료 환경에 자동 반영
     버전 관리      : git 이력으로 변경 추적
     설치 한 줄     : zip 풀기 단계 없음

  📊 적합도
     소요 시간    : 30분 (1회성)
     적합 규모    : 3~10명, git에 익숙한 팀
     장점        : 자동 업데이트, 버전 추적, 설치 더 간결
     단점        : git 사용 학습 필요
```

**발표 노트 (1분 30초)**:
> "2단계, 플러그인 git 공유입니다. git을 쓸 수 있는 분이라면 이 단계가 훨씬 편합니다.
>
> 1단계랑 plugin.json은 똑같이 만들고, 그걸 git 저장소에 push 합니다. 그러면 동료는 '/plugin install' 한 줄에 git url만 붙이면 끝이에요. zip 풀기 단계 없습니다.
>
> 1단계 zip과의 결정적 차별점은 — 자동 업데이트입니다. 여러분이 skill을 수정해서 git push 하면 동료 환경에 자동으로 반영돼요. 그리고 git 이력으로 누가 언제 뭘 바꿨는지 추적도 됩니다.
>
> 개발자 분들이나 git에 익숙한 팀이면 이 단계로 가시는 게 정답입니다. 비개발자 분들은 1단계 zip이 더 편하시고요. 본인 환경에 맞게 선택하세요."

**시각 표현 메모**:
- git 명령어 + 설치 한 줄 박스.
- ⭐ 차별점 박스로 1단계와의 차이 명확히.
- "자동 업데이트" 강조.

## 슬라이드 7-4 — 3단계: 팀 마켓플레이스 (1분 15초)

**핵심 메시지**: "marketplace.json 한 번 만들면 팀 전체가 새 플러그인을 자동 발견."

**본문 콘텐츠**:

```
3단계 — 팀 마켓플레이스 구축 (전사 표준화)

  ───────────────────────────────────────────

  팀 공용 git 저장소에 marketplace.json 1회 생성

  {
    "name": "BDI Plugins",
    "plugins": [
      { "name": "research-team-plugin",
        "source": "github",
        "repo": "bdi/research-plugin" },
      { "name": "report-tone-plugin",
        "source": "github",
        "repo": "bdi/report-tone" }
    ]
  }

  ───────────────────────────────────────────

  팀원 한 번만 등록:
  /plugin marketplace add <git-url>

  → 이후 새 플러그인 추가 시 모든 팀원이 자동 발견

  ───────────────────────────────────────────

  📊 적합도
     소요 시간    : 초기 4시간 (1회) + 신규 플러그인당 30분
     적합 규모    : 10명 이상 (팀·부서·전사 표준화)
     장점        : 표준 플러그인 카탈로그, 자동 발견·업데이트
     단점        : 초기 셋업 부담
```

**발표 노트 (1분 15초)**:
> "3단계, 팀 마켓플레이스입니다. 부서나 전사 단위로 표준화할 때 쓰는 방식이에요.
>
> 마켓플레이스 저장소 하나에 marketplace.json을 두면, 거기에 등록된 플러그인 목록을 팀원 모두가 한 번 등록하고 자동으로 발견합니다. 새 플러그인이 추가되면 팀원에게 자동 알림이 가요. 10명 이상 BDI처럼 연구원 전체가 공유한다면 이 단계로 가는 게 맞아요.
>
> 정리하면 — 0단계는 한 파일, 1단계는 묶음 zip, 2단계는 묶음 git, 3단계는 전사 카탈로그. 본인 규모와 git 친숙도에 맞게 시작하시면 됩니다. 그리고 다음으로 — 제가 직접 매일 쓰는 플러그인 하나 추천하고 마무리하겠습니다."

**시각 표현 메모**:
- json 코드블록 + 등록 명령 박스.
- 마지막 줄에서 챕터 8(superpowers)로 자연 연결.

## 챕터 7 합산

- 슬라이드 4장 (git 공유를 별도 단계로 분리)
- 발표 노트 합산: 45 + 90 + 90 + 75 = 300초 = 5분 정확
- 4단계 구조:
  - **0단계**: 파일 한 개 공유 — 비개발자, git 불필요
  - **1단계**: 플러그인 zip 공유 — 묶음, git 불필요
  - **2단계**: 플러그인 git 공유 — 묶음, 자동 업데이트, git 사용
  - **3단계**: 팀 마켓플레이스 — 전사 표준화
- git 우회 경로(0·1단계)와 git 활용 경로(2·3단계)를 모두 명확히 제시

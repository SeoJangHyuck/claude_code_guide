# 챕터 5: .claude 폴더 구조 심화 (9슬라이드 / 10분)

> 톤 가이드: 청중에 비개발자 비중이 큼. 우선순위 — **CLAUDE.md → skills → rules**가 메인. settings.json/hooks는 후순위(짧게). agents·MCP·plugins은 1분 내 압축.

> 모든 코드 예시는 2026-05 기준 Claude Code 공식 형식 준수.

## 슬라이드 5-1 — 전체 폴더 트리 (1분 30초)

**핵심 메시지**: ".claude 폴더 = Claude Code의 작업 공간. 7가지 요소가 들어간다."

**본문 콘텐츠**:

```
프로젝트/
├── .claude/
│   ├── CLAUDE.md                   ① 프로젝트 헌법 (항상 읽힘)
│   ├── skills/                     ② 재사용 가능한 절차서
│   │   └── 내-skill-이름/
│   │       └── SKILL.md
│   ├── rules/                      ③ 경로별 조건 규칙
│   │   └── 내-rule-이름.md
│   ├── settings.json               ④ 권한·환경·hooks 설정
│   ├── settings.local.json         (개인용, gitignored)
│   ├── agents/                     ⑤ 격리된 작업자
│   │   └── 내-agent-이름/
│   │       └── AGENT.md
│   ├── .mcp.json                   ⑥ 외부 시스템 연결
│   └── .claude-plugin/             ⑦ 위 ②~⑥을 묶은 패키지
│       └── plugin.json
└── (그 외 일반 프로젝트 파일)
```

> 참고: CLAUDE.md는 `./.claude/CLAUDE.md` 또는 `./CLAUDE.md` (루트) 둘 다 공식 지원.

**발표 노트 (1분 30초)**:
> "Claude Code의 모든 커스터마이징은 .claude라는 폴더 한 군데에 모입니다. 여기에 7가지 요소가 들어가요. 하나씩 짧게 보겠습니다.
>
> 우선순위가 중요합니다. ① 프로젝트 헌법 CLAUDE.md, ② 재사용 절차 skills, ③ 경로별 규칙 rules — 이 셋이 가장 자주 손대는 것들이에요. 비개발자 분도 이 셋만 알아도 90%입니다.
>
> ④ 권한·hooks 설정, ⑤ 격리 작업자 agents, ⑥ 외부 시스템 연결 MCP — 이건 좀 더 익숙해진 다음에. ⑦ plugin은 ②~⑥을 한 묶음으로 동료에게 나눠줄 때 씁니다.
>
> 한 가지 팁만 — 이 폴더 통째로 git에 올리면, 동료가 그대로 받아 씁니다. 사내 공유의 출발점이 바로 이 폴더예요."

**시각 표현 메모**:
- 폴더 트리 한 화면. 7요소 번호·강조색.
- ①②③ 강한 강조색 (메인 3종), ④⑤⑥⑦은 옅은 색.
- 우측에 "이 폴더 통째 git에 올리면 = 사내 공유" 작은 박스.

## 슬라이드 5-2 — ① CLAUDE.md (1분 30초)

**핵심 메시지**: "프로젝트의 헌법. 한 번 적어두면 모든 대화에 자동 적용."

**본문 콘텐츠**:

```
CLAUDE.md  =  프로젝트의 헌법

  • 위치: ./.claude/CLAUDE.md  (또는 ./CLAUDE.md  — 둘 다 공식 지원)
  • 자동 로드: 세션 시작 시 항상
  • 권장 분량: 200줄 이하
  • 무엇을 적나: 항상 적용해야 하는 규칙·명령어·컨벤션

  ──────────────────────────────────────────

  # 정책 연구 작업 가이드

  ## 글쓰기 컨벤션
  - 보고서는 ~다 체로
  - 출처는 항상 파일 경로 + 줄번호 명시
  - 분량 본문 2장 이내

  ## 디렉토리 구조
  - reports/ : 정기 보고서·다이제스트
  - data/    : 원본 엑셀·PDF
  - drafts/  : 발송 전 초안
```

**발표 노트 (1분 30초)**:
> "첫 번째, CLAUDE.md입니다. .claude 폴더 안에 그냥 마크다운 파일 하나 두는 거예요 (프로젝트 루트도 공식 지원, 취향 차이). 세션을 시작할 때마다 Claude Code가 알아서 읽습니다. 별도 호출 없어요.
>
> 여기에 뭘 적나? 매번 똑같이 알려줘야 하는 것들요. 빌드 명령어, 우리 팀 글쓰기 컨벤션, 디렉토리 의미 같은 거. 적어 놓으면 다음부터 안 알려줘도 알아서 따릅니다.
>
> 한 가지 주의 — 200줄이 넘어가면 길어요. 다단계 절차나 가끔 필요한 작업은 다음에 볼 'skill'로 빼는 게 좋습니다. 그리고 특정 폴더에만 적용되는 규칙은 그 다음에 볼 'rules'로 빼고요."

**시각 표현 메모**:
- 상단: 정의 박스
- 하단: 실제 CLAUDE.md 예시 (가상의 연구원 프로젝트 톤)

## 슬라이드 5-3 — ② skills (1분 30초)

**핵심 메시지**: "재사용 가능한 절차서. description이 자동 발동의 키."

**본문 콘텐츠**:

```
.claude/skills/<이름>/SKILL.md  =  다단계 절차서

  ──────────────────────────────────────────

  ---
  name: weekly-policy-digest
  description: 매주 부산연구원 등 정책기관 4곳의
               신규 자료를 수집·요약해 다이제스트로 만든다.
               월요일 또는 "주간 동향" 요청 시 발동.
  allowed-tools: Bash, Read, Write, WebFetch
  ---

  ## 절차
  1. 4개 기관 사이트에서 지난 주 신규 자료 목록 수집
  2. 각 자료를 다운로드해 요약 (subagent 4개 병렬)
  3. reports/주간동향-{날짜}.md 로 저장
  ──────────────────────────────────────────

  💡 핵심: description = Claude가 자동 발동할지 판단하는 키
       → "언제 쓰는지"를 명확히 적어야 잘 호출됨
```

**발표 노트 (1분 30초)**:
> "두 번째, skill입니다. 오늘 발표의 핵심 도구예요. 다단계 작업을 한 번 정리해두는 거예요.
>
> 보시면 위쪽에 frontmatter라는 메타데이터가 있고, 그 아래에 절차가 있습니다. 이 frontmatter의 'description'이 가장 중요한데요, Claude가 사용자 요청을 듣고 '이 skill을 자동으로 부를까?' 결정하는 키가 됩니다.
>
> 예시는 '매주 정책기관 동향 다이제스트' skill입니다. 한 번 만들어 두면 다음 주에 '주간 동향 정리해줘' 한 마디만 하면 알아서 4기관 수집·요약·저장까지 합니다.
>
> CLAUDE.md와 차이는 — CLAUDE.md는 '항상' 읽히고, skill은 '필요할 때만' 호출됩니다. 매번 적용할 게 아니면 skill로 빼는 게 맞아요."

**시각 표현 메모**:
- frontmatter와 본문을 구분선으로 분리.
- 하단 💡 박스로 핵심 강조.

## 슬라이드 5-4 — ③ rules (1분)

**핵심 메시지**: "경로별 조건 규칙. paths 글로브에 매칭되는 파일을 다룰 때만 자동 로드."

**본문 콘텐츠**:

```
.claude/rules/<이름>.md  =  경로별 조건 규칙

  ──────────────────────────────────────────

  .claude/rules/reports.md

  ---
  paths:
    - "reports/**/*.md"
  ---

  보고서 작업 시 적용 규칙
  (reports/ 폴더의 .md 파일을 다룰 때 자동 로드)

  - 항상 "~다" 체로 작성
  - 모든 인용에 파일 경로 + 줄번호 명시
  - 분량 2장 이내
  - "결론" 섹션 필수

  ──────────────────────────────────────────

  CLAUDE.md vs rules
  ────────────────────────────────────────
  CLAUDE.md  →  프로젝트 전체, 매 세션 자동 로드
  rules      →  paths 글로브에 매칭될 때만 로드
                → 컨텍스트 절감 + 영역별 분리
```

**발표 노트 (1분)**:
> "세 번째, rules입니다. CLAUDE.md와 비슷하지만 결정적 차이가 있어요. CLAUDE.md는 프로젝트 전체에 매 세션마다 자동 로드되는 헌법이라면, rules는 위쪽 'paths' 한 줄에 매칭되는 파일을 다룰 때만 자동 로드됩니다.
>
> 예시처럼 'reports 폴더의 마크다운 파일을 다룰 때만 이 규칙이 발동된다' 이런 식이에요. 보고서 작업할 때만 적용되는 글쓰기 톤, 코드 폴더에서만 적용되는 컨벤션 — 영역별로 규칙을 쪼개 두면 컨텍스트도 가벼워지고 CLAUDE.md도 비대해지지 않습니다."

**시각 표현 메모**:
- 상단: rules 정의 + 실제 예시(frontmatter `paths` 강조)
- 하단: CLAUDE.md vs rules 두 줄 비교 박스
- 좌측 하단 작게 — "출처: code.claude.com/docs/en/memory.md"

## 슬라이드 5-5 — ④ settings.json + hooks (1분)

**핵심 메시지**: "권한 자동 승인·차단 + 도구 실행 전후 자동 트리거. 한 번 설정 후 잊고 사는 안전장치."

**본문 콘텐츠**:

```
.claude/settings.json  =  권한·hooks 설정 (안전장치)

  ──────────────────────────────────────────

  {
    "permissions": {
      "allow": ["Read(./**)", "Write(./reports/**)"],
      "deny":  ["Read(./.env)", "Bash(rm -rf *)"]
    },
    "hooks": {
      "PostToolUse": [{
        "matcher": "Write",
        "hooks": [{ "type": "command",
                    "command": "bash demo_assets/busan_ppt/verify_report.sh" }]
      }]
    }
  }

  ──────────────────────────────────────────

  💡 비개발자도 의미 있는 이유
       위험한 명령은 차단 · 저장 직후 자동 검증
       → "한 번 설정해두면 사고 안 나는 가드레일"
```

**발표 노트 (1분)**:
> "네 번째, settings.json입니다. 비개발자 분께 이건 한 마디로 '안전장치'예요. 위험한 명령은 차단하고, 매번 잊어버리는 검증 작업은 자동 실행되게 하는 가드레일.
>
> 예를 들어 '.env 파일은 절대 읽지 마라', '보고서를 저장한 직후엔 자동으로 검증 스크립트 돌려라' — 이런 식으로 한 번만 설정해두면 그 다음부터 신경 안 써도 됩니다. 6장 시연에서 이 hook이 실제로 출처 없는 슬라이드를 저장 차단하는 모습을 보여드릴 거예요. 디테일은 익숙해진 다음에 손대셔도 늦지 않아요."

**시각 표현 메모**:
- 코드 짧게 유지 (위 분량이 최대치).
- 💡 박스로 비개발자 의미 강조.

## 슬라이드 5-6 — ⑤ agents (50초)

**핵심 메시지**: "격리된 작업자. 큰 탐색·병렬에 적합."

**본문 콘텐츠**:

```
.claude/agents/<이름>/AGENT.md  =  격리 작업자

  ──────────────────────────────────────────

  ---
  name: site-crawler
  description: 한 사이트의 자료 목록을 수집·요약한다.
  tools: WebFetch, Read, Write
  model: sonnet
  ---

  당신은 정책기관 사이트 수집 전문가입니다.
  지정된 URL에서 지난 주 신규 자료를 모두 가져와
  각 자료의 핵심 3줄을 정리해 반환합니다.

  ──────────────────────────────────────────

  Skill vs Agent
  ──────────────────────────
  Skill   = 메인 대화에 합류, 결과 그대로 노출
  Agent   = 별도 컨텍스트에서 작업, 결과만 요약 반환
            → 큰 탐색 / 병렬 처리에 적합
```

**발표 노트 (50초)**:
> "다섯 번째, agent입니다. skill과 비슷해 보이지만 별도 작업 공간에서 독립적으로 일하다 결과만 반환합니다. 정책기관 4곳을 동시에 수집해야 할 때처럼 병렬 처리·큰 탐색에 적합해요. 한 줄 기준 — 큰 탐색·병렬·메인 대화 보호가 필요하면 agent, 그냥 절차 재사용이면 skill."

**시각 표현 메모**:
- 상단: agent 정의 예시
- 하단: skill vs agent 한 줄 비교 박스

## 슬라이드 5-7 — ⑥ MCP 서버 (50초)

**핵심 메시지**: "외부 시스템(GitHub·Slack·DB 등)과 잇는 다리."

**시각 표현 메모**:
- json 코드블록 짧게.
- 하단에 대표 MCP 6개 한 줄 나열.

## 슬라이드 5-8 — ⑦ plugins (50초)

**핵심 메시지**: "위 6가지를 한 패키지로 묶어 한 번에 배포·설치."

**본문 콘텐츠**:

```
.claude/.claude-plugin/plugin.json  =  자산 묶음 패키지

  ──────────────────────────────────────────

  {
    "name": "research-team-plugin",
    "version": "1.0.0",
    "description": "정책 연구원 업무 자동화 묶음",
    "author": { "name": "BDI 데이터팀" }
  }

  ──────────────────────────────────────────

  plugin이 묶을 수 있는 자산
  ──────────────────────────────────────
  ✓ skills   ✓ rules    ✓ agents
  ✓ hooks    ✓ MCP      ✓ commands

  → 동료는 한 줄로 설치
     /plugin install research-team-plugin
```

**발표 노트 (50초)**:
> "일곱 번째, plugin입니다. 지금까지 본 ②부터 ⑥까지를 한 묶음으로 패키징해서 동료가 한 줄로 설치하게 만드는 거예요. .claude-plugin 폴더에 plugin.json 하나만 추가하면 됩니다. 참고로 우리 데모 프로젝트 .claude-plugin에 plugin.json이 하나 있는데 그건 제가 받아 쓴 외부 슬라이드 플러그인이고, 위 슬라이드 예시는 '우리가 만든 자산을 묶으면' 어떻게 되는지 가상 예시입니다. 사내 공유의 가장 깔끔한 방식 — 챕터 7에서 더 자세히 보겠습니다."

**시각 표현 메모**:
- json 코드블록 짧게.
- 하단에 묶을 수 있는 자산 6개 + 설치 명령 한 줄.

## 슬라이드 5-9 — 의사결정 트리 (1분)

**핵심 메시지**: "이걸 만들 때 어디에 넣지? — 한 장으로 정리."

**본문 콘텐츠**:

```
   "내가 뭔가를 자동화·재사용하고 싶다"
                  │
                  ▼
   ┌──────────────────────────────────────┐
   │ 항상 자동 적용되어야 하는가?           │ ── YES ──▶  CLAUDE.md
   └────────────┬─────────────────────────┘
                │ NO
                ▼
   ┌──────────────────────────────────────┐
   │ 특정 폴더·파일에서만 적용?             │ ── YES ──▶  rules
   └────────────┬─────────────────────────┘
                │ NO
                ▼
   ┌──────────────────────────────────────┐
   │ 반복 태스크 · 워크플로우를 재사용?      │ ── YES ──▶  skill
   └────────────┬─────────────────────────┘
                │ NO
```

Chapter 4 — 도구·제품 비교 2 / 3
결정적 차이 4가지
①번 축(답변 품질)은 다 강하다. ②~④번 축에서 갈린다 — 일하는 흐름 순서로 4가지.
① 시작
터미널 없이 VSCode 확장 한 줄로 시작

검은 터미널 X — 파일 트리·메뉴·마우스 그대로

확장 설치 = Claude Code 연동 완료

VSCode 마켓 확장도 같이 씀 — Markdown All in One · vscode-pdf · Code Spell Checker (각 5M+ 다운로드)

→ 5장 진입 가이드 + 6장 시연도 VSCode 확장으로

② 작업
작업을 클라우드 연동 없이 로컬에서

파일이 내 PC에서 그대로 처리 — 클라우드 업로드 자체가 없음

채팅형 도구(ChatGPT·Gemini·Claude 웹)는 학습 옵트아웃은 되지만, 작업 시작 시 파일은 클라우드로 올라감

→ 5장 settings.json에서 권한·자동 가드 설정

③ 자산화·검증
같은 작업을 매주 다시 짜지 않고, 환각이 박히기 전에 막는다

Skills — 한 번 짜둔 절차를 매주 호출 (반복 분석·자료 정리)

Subagents — 긴 작업·여러 자료 병렬 처리 (격리된 컨텍스트)

Hook 29개 이벤트 — 파일 저장 직전·도구 호출 전 자동 검증, settings.json 한 줄로 동작

→ 5장 skills · agents · hooks

④ 공유
동료가 만든 절차를 git 또는 plugin으로 받아쓴다

git — .claude/ 폴더 통째로 공유 (절차·권한·외부 연결까지)

Plugin — 패키지로 묶어 marketplace 배포, 한 줄로 설치

MCP 오픈 표준 — 다른 도구가 만든 연결도 그대로 흡수

GitHub OSS 자산도 동일 — awesome-claude-code · claude-code-skills · plugins

→ 7장 사내 공유 flow (파일 → zip → git → marketplace 4단계)

Chapter 5 — .claude 폴더 구조 1 / 9
.claude 폴더 = Claude Code의 작업 공간
7가지 요소가 모인다. 우선순위는 ① CLAUDE.md → ② skills → ③ rules (메인 3종).
프로젝트/ ├── .claude/ │ ├── CLAUDE.md ① 프로젝트 헌법 (항상 자동 로드) │ ├── skills/ ② 재사용 가능한 절차서 │ │ └── 내-skill-이름/ │ │ └── SKILL.md │ ├── rules/ ③ 경로별 조건 규칙 │ │ └── 내-rule-이름.md │ ├── settings.json ④ 권한·환경·hooks 설정 │ ├── settings.local.json (개인용, gitignored) │ ├── agents/ ⑤ 격리된 작업자 │ │ └── 내-agent-이름.md │ └── .claude-plugin/ ⑦ 위 ②~⑥을 묶은 패키지 │ └── plugin.json ├── .mcp.json ⑥ 외부 시스템 연결 (프로젝트 루트) └── (그 외 일반 프로젝트 파일)
CLAUDE.md는 ./.claude/CLAUDE.md 또는 ./CLAUDE.md (루트) 둘 다 공식 지원. 이 폴더 통째로 git/zip 공유하면 동료가 동일 환경 즉시 사용.

정의

항상 적용되는 규칙·컨벤션

  • 위치: .claude/CLAUDE.md (또는 루트 ./CLAUDE.md) — 둘 다 공식 지원
  • 로드: 세션 시작 시 자동

정의

  • MCP = Model Context Protocol (Anthropic이 공개한 표준)
  • 역할: Claude가 외부 서비스 API를 직접 호출할 수 있도록 인증·요청·응답을 중간에서 처리

  • 자동 호출: 등록된 MCP 중 Claude가 작업 맥락 보고 필요한 것만 호출 — 매번 지시할 필요 X

Chapter 7 — 사내 공유 flow 2 / 4
2단계 — 플러그인 zip 공유
여러 자산(skill·agent·hook)을 한 묶음으로. 여전히 git 불필요.
2
플러그인 zip 공유 git 불필요
당신 — 만들기
① 플러그인 디렉토리 (이름·위치 자유) 
my-plugin/
├── .claude-plugin/plugin.json   ← 필수
├── skills/<이름>/SKILL.md
├── agents/<이름>.md
└── hooks/hooks.json              (선택)
② plugin.json — 4필드 
{ "name": "research-team-plugin",   ← 호출 prefix
  "description", "version", "author" }
③ 로컬 검증 → zip 압축 → 메신저 전달 claude --plugin-dir ./my-plugin
동료 — 받기
① zip 압축 풀기
② 두 줄로 설치 /plugin marketplace add /풀어둔/경로 /plugin install research-team-plugin → 호출 시 네임스페이스 prefix로: /research-team-plugin:skill-name (prefix = plugin.json의 name 필드)
적합 규모
3~10명, git 어려운 팀
장점
묶음 설치, 네임스페이스 충돌 방지
단점
자동 업데이트 X (수정 시 zip 재전달)
⚠️ 흔한 실수: skills/·agents/·hooks/를 .claude-plugin/ 안에 넣지 말 것 — plugin.json '옆'에 같은 레벨로.  📖 Plugins 가이드

Chapter 7 — 사내 공유 flow 3 / 4
3단계 — 플러그인 git 공유
한 줄 설치 + 자동 업데이트. git에 익숙한 팀에 적합.
3
플러그인 git 공유 자동 업데이트
당신
1단계와 같은 플러그인 디렉토리를 git 저장소에 push cd my-plugin/ git init && git remote add origin <url> git add . && git commit -m "share plugin" git push -u origin main
동료
한 줄로 설치 /plugin install <name>@<git-url> ※ 당신이 push 하면 동료 환경에 자동 반영
적합 규모
3~10명, git 사용 팀
장점
자동 업데이트, 버전 추적
단점
git 사용 학습 필요
핵심 형식: git URL은 HTTPS(https://github.com/팀/플러그인.git) 또는 SSH(git@github.com:팀/플러그인.git) 모두 가능. private repo는 토큰·SSH 키 사전 설정 필요.
📖 참고: docs.claude.com — Plugins 가이드 (설치 섹션)

Chapter 7 — 사내 공유 flow 4 / 4
4단계 — 팀 마켓플레이스
marketplace.json 한 번 만들면 팀 전체가 새 플러그인을 자동 발견.
4
팀 마켓플레이스 전사 표준화
당신 (1회 셋업)
팀 공용 git 저장소에 marketplace.json 작성 — 우리 팀이 쓰는 플러그인 카탈로그: { "plugins": [ {"name": "busan-report", "source": "git@...:plugins/busan", "version": "1.0"}, {"name": "thursday-mail", "source": "git@...:plugins/mail", "version": "2.1"} ] } → 신규 플러그인은 항목만 추가 · 모든 팀원이 자동 인지
팀원 (최초 1회 + 평소)
/plugin marketplace add <git-url> ↑ 1회 — 우리 팀 카탈로그 구독 /plugin install busan-report ↑ 카탈로그에서 필요한 것만 골라 설치 ※ marketplace.json이 갱신되면 새 플러그인이 자동으로 카탈로그에 노출 — 팀원은 install 한 줄만 추가
적합 규모
10명+ (팀·부서·전사)
장점
표준 카탈로그, 자동 발견·업데이트
단점
초기 셋업 부담
3단계와의 차이: 3단계는 플러그인 N개를 각각 /plugin install A@git-a, B@git-b … N번 설치 → 새 플러그인 추가 시 팀원에게 git URL을 매번 공유해야 함. 4단계는 카탈로그 1번 add → 이후 카탈로그 갱신만으로 팀원에게 자동 노출.

Chapter 8 — 추천 플러그인: superpowers 1 / 1
superpowers — 검증을 강제하는 플러그인
AI가 빠르게 일할수록 건너뛰기 쉬운 4가지를, 단계마다 강제로 멈추고 챙기게 한다.
🧠 brainstorming
만들기 전 강제 게이트
"이 결과가 맞나요?", "빠뜨린 케이스 없나요?" — 요구사항·의도를 먼저 명확화
→ 잘못된 방향으로 시작하는 시간 차단
🧪 test-driven-development
구현 전 강제 게이트
"무엇을 통과해야 완성인가" — 검증 기준부터 먼저 정의
→ "다 했다"의 기준이 명확해짐
🔍 systematic-debugging
버그 발견 시 강제 게이트
즉시 수정 금지 — 원인·재현·유사 사례 진단부터
→ 임시방편 수정의 연쇄 사고 방지
✅ verification-before-completion
완료 선언 직전 강제 게이트
체크리스트 자동 호출 — 출처·검증·테스트 통과 확인
→ 환각·미완성 결과를 자동 감지
💡 왜 부산연구원 청중에게 추천하나
설문 1순위 — AI 결과물 신뢰성·환각·검증 부담 (60%). superpowers의 정체성이 "검증 없이 끝내지 못하게 만드는 강제 게이트" — 같은 고민의 정면 답.
설치 (Anthropic 공식 마켓플레이스) /plugin install superpowers@claude-plugins-official 또는 자체 마켓플레이스 (관련 플러그인 포함): /plugin marketplace add obra/superpowers-marketplace → /plugin install superpowers@superpowers-marketplace · 📖 github.com/obra/superpowers
